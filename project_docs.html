Why intraday is v1, not cut
Intraday is a separate hypothesis: the market lags observed temperature data in the 12pm-2pm window because retail bettors anchor to the morning forecast and check back infrequently. This is distinct from the morning mispricing hypothesis and could hold even if the morning model has no edge. It stays in v1 because it requires the morning model infrastructure to exist first, and because validating it independently needs live prices -- not paper trades. High conviction means build it in v1, not v0.

ANALYSIS NOTES (documentation only, not fields)
The morning_model.py 7am run writes rows with snapshot_type = 'morning'. These are the ONLY rows used for v0 evaluation metrics (Brier score, permutation test, paper P&L).

collector.py writes rows with snapshot_type = 'intraday'. These enable edge decay analysis, information timing tests, and Q4 decomposition but are excluded from v0 gate evaluation.

event_date is the primary key concept. A single event_date (e.g. 2026-04-02) will have one morning snapshot (6 bracket rows from the 7am run) and potentially 20+ intraday snapshots (6 bracket rows each from collector.py running 30-60min intervals from market open through settlement).

When the morning model runs at 7am on April 2nd, it should log event_date = 2026-04-02. If it also captures tomorrow's market (event_date = 2026-04-03), those rows get forecast_lead_hours ~31 and can be analyzed separately to measure whether longer-lead forecasts have larger or different edge patterns.

The implementer should use either: (a) One wide table with all fields, or (b) Three tables: event_days (keyed on event_date), snapshots (keyed on event_date + snapshot_ts), brackets (keyed on event_date + snapshot_ts + bracket_label). Option (b) avoids repeating day-level data across bracket rows and snapshot metadata across bracket rows. Choose whichever is simpler to implement and query.

Working hypothesis -- direction unknown
What we are testing
Hypothesis updated. Previous version assumed tails are underpriced. That assumption is now invalidated by real-world signal. v0 is direction-agnostic -- it measures mispricing across all brackets and lets the data determine direction.
Core hypothesis (updated 2026-03-30): Kalshi NYC temperature markets systematically misallocate probability mass across brackets relative to probabilities implied by p10/p50/p90 NBM guidance. Research suggests the most likely direction is favorite-longshot bias -- tail brackets overpriced, middle brackets underpriced -- consistent with the platform-wide pattern documented in Bürgi et al. (2026) and the anecdotal participant signal. However, weather markets may behave differently due to bot activity and ensemble model availability. Direction remains an output of v0 data.
Three questions -- in order
Q1 -- Does mispricing exist at all?
Test: Does model_prob minus market_price show a consistent directional pattern across brackets? Not noise -- a repeatable sign.

How to answer: After 20-30 days, plot edge (model_prob minus market_price) for every bracket-day pair. Are middle brackets consistently positive or negative? Are tails? Is there a pattern at all?

Status: Untested. This is the only question v0 needs to answer before anything else.
Q2 -- What direction? (only after Q1 is yes)
This is now the most important v0 output. Specifically:

Are middle brackets (near p50) systematically cheap or rich vs model?
Are tail brackets (far from p50) systematically cheap or rich?
Is there asymmetry between upper and lower tails?

Do not answer this question before Q1. If there is no consistent pattern in Q1, direction analysis is noise.
Q3 -- Is mispricing conditional? (secondary, after Q1 + Q2)
Is the pattern stronger in certain setups -- high vs low spread (p90-p10), record proximity, large forecast shifts? These are secondary analyses, not core assumptions. Test only after Q1 and Q2 have answers. High uncertainty is a candidate modifier, not a requirement.
Q4 -- INFORMATION TIMING (secondary, from market structure research)
How quickly does the market incorporate new forecast and observational data? Research identifies five bot types reacting to DSM releases, 6-hourly max readings, ASOS one-minute data, and CLI reports within seconds. If Kalshi prices fully reflect the 06Z GFS ensemble by 7am, the morning model may be measuring noise rather than mispricing.

Test from v0 data: using intraday price and model_prob snapshots from collector.py, decompose edge change into two components -- did model_prob move (NBM updated) or did market_price move (traders repriced)? If the market moves toward the 7am model_prob, that is evidence of real morning mispricing. If model_prob moves toward the morning market_price, the market was ahead of the model.

Also measure: correlation between morning edge (7am model_prob minus market_price) and subsequent price movement direction. If morning edge predicts the direction prices move during the day, the market was slow to incorporate information the model captured.

This requires collector.py to be running. It does not affect v0 model logic or evaluation gates.
Three evaluation layers -- failure at each means something different
Do not treat a v0 failure as a single "no edge" result. The evaluation has three distinct layers. Diagnosing which one broke determines the next step.
Layer 1 -- Is the probability model any good?
Question: Does model_prob (from zone interpolation) accurately reflect the true probability of each outcome?

Metric: Brier score. For each bracket-day pair, compute (model_prob - outcome)^2. Average across all logged rows. Compare against the naive normal distribution baseline using the same metric.

Failure means: The probability conversion is broken -- ERA5 proxy is wrong, interpolation has a systematic flaw, or nbm_bias is badly estimated. Fix the model before testing mispricing.
Layer 2 -- Is the market mispriced relative to the model?
Question: Is model_prob - market_price (edge) consistently non-zero in a repeatable direction?

Metric: Mean edge and directional persistence. For each bracket position (tail vs middle, measured as distance from p50 in sigma units), compute mean edge across days. Also compute the fraction of days where edge has the same sign as the mean -- this is directional persistence.

Failure means: The model may be well-calibrated but the market is also well-calibrated. No exploitable gap exists at the bracket level. Not a model failure -- a market efficiency finding.
Layer 3 -- Is the gap economically tradable?
Question: After fees, spreads, and execution friction, is hypothetical P&L positive?

Metric: After-cost paper P&L across all paper-trade days. Model a realistic round-trip cost (entry spread + Kalshi fee) and subtract from gross edge. Compare against zero and against baseline 1 (NBM p50 bracket).

Failure means: Edge exists but is too small to capture. This is a microstructure problem, not a model or market problem. Could still be salvageable if edge is concentrated in specific setups.

Noise test -- predeclared decision rule
This must be specified before v0 ends, not after
Method: Permutation test on mean edge. Shuffle the outcome labels (actual_max_f) across days, recompute mean edge for each bracket position 1000 times, and check where the observed mean edge falls in that null distribution.

Decision rule: If observed mean edge falls above the 90th percentile of the permuted distribution (p < 0.10, one-tailed), edge is unlikely to be noise. This threshold is intentionally lenient given the sample size -- it filters obvious noise, not marginal cases.

Why permutation, not t-test: Edge values are not normally distributed and may be serially correlated across days. Permutation is distribution-free and more honest for this data structure.

If this test is not pre-registered before v0 ends, the v0/v1 gate cannot be evaluated without post-hoc bias risk.
Baselines -- with predeclared comparison metric
Comparison metric for all baselines: after-cost paper P&L. A baseline "wins" if its hypothetical after-cost P&L over the v0 sample exceeds the model's. This is the single metric used for all comparisons -- no mixing of calibration metrics and P&L metrics across baselines.
1. NBM p50 bracketMUST BEAT
Bet the bracket containing the NBM 50th percentile every day at the prevailing market price. Primary economic baseline. If the model cannot beat this, bracket conversion adds nothing.
2. Naive normal distributionMUST BEAT
Same interpolation logic but using scipy norm.fit() instead of asymmetric zones. Tests whether the tail-width fix earns its complexity. Also compared on Brier score, not just P&L.
3. Highest market-probability bracketMUST BEAT
Buy whichever bracket Kalshi prices highest each day. No model needed. Tests whether the model adds anything over betting the crowd favourite.
4. Simplification check (not an independent baseline)DIAGNOSTIC
Take only the bracket with the largest absolute edge each day. This is derived from the model itself so it is not an independent baseline -- it is a simplification check. If it matches full model P&L, no classification logic beyond "biggest gap" is adding value.

Do not build directional logic into v0. No bias toward tails. No bias toward middles. No flag for "positive edge only." Log all brackets, all edge values, positive and negative. The logging structure enables post-hoc analysis. The v0 model has no opinion on direction.

Pre-registered sub-hypotheses (written before v0 data)
FLB TEST
Favorite-longshot bias predicts that Kalshi weather tail brackets (market_price below 0.15) are systematically overpriced and middle brackets (market_price above 0.50) are systematically underpriced relative to true probabilities.

Test: After 20-30 days of v0 data, partition all bracket-day pairs by market_price into two groups: above 0.50 and below 0.15. Compute mean edge (model_prob minus market_price) for each group. FLB predicts mean edge is negative for the cheap group and positive for the expensive group.

This is a secondary analysis. It does not affect v0 model logic, logging, or the three primary evaluation layers. Pre-registered 2026-03-29.

Sources: Bürgi, Deng & Whelan (2026) “Makers and Takers” CEPR DP 20631; anecdotal participant report (see Evidence Log above).
EDGE DECAY TEST
If morning mispricing is real, edge should decay through the day as the market incorporates new information. If edge is model error, it should persist or reverse as the NBM updates toward what the market already knew.

Test: Using hourly snapshots from collector.py, compute edge at each timestamp. Plot mean absolute edge vs time of day. If edge decays monotonically from 7am to 3pm, the morning model captures real information the market is slow to price. If edge is flat or increases, the model is not capturing actionable mispricing.

Also decompose: on days where 7am edge was positive on a bracket, did model_prob decrease toward market_price (NBM corrected toward market) or did market_price increase toward model_prob (market corrected toward model)? The direction of convergence identifies whether the model or the market was 'right' on average.

Requires collector.py hourly model_prob snapshots. Pre-registered 2026-03-30.

collector.py is a run-and-exit script, not a long-running daemon. Plain crontab on the VM is the simplest scheduler: no extra dependencies, no process manager configuration, easy to debug. Each cron invocation runs the script, writes to SQLite, and exits. If a run fails, the next invocation 30 minutes later tries again -- at most one snapshot is lost. Shell wrapper script handles virtualenv activation and log routing. Health check via heartbeat file, checked by a separate cron job every 2 hours. morning_model.py remains a manual 7am run in v0 -- it is not added to cron.

2026-03-30 — Collector captures all open event dates, not just today
Kalshi opens tomorrow's temperature market at 10am ET the day before. By mid-morning, two event dates are typically trading simultaneously. Collector now captures brackets for all open KXHIGHNY markets at each snapshot. Benefits: (1) early-market prices for tomorrow are the thinnest and most likely to be mispriced, (2) paired observations at different forecast lead times enable lead-time analysis from the same model run, (3) effectively doubles the sample size for trend analysis. The morning_model.py 7am evaluation run still targets today's event_date only. Multi-day capture is collector-only.

2026-03-30 — Kalshi bracket boundary convention confirmed from CFTC filings
NHIGH contract terms (CFTC filing) confirm: tail brackets use strict inequality ('greater than 69' excludes exactly 69, 'less than 62' excludes exactly 62). Between brackets are inclusive of both endpoints ('62-63' includes both 62 and 63). NWS CLI reports integer °F only, so the outcome space is discrete with no edge cases.

For the continuous probability model, a half-degree continuity correction makes boundaries contiguous:
  <62    integrates  -inf  to 61.5
  62-63  integrates  61.5  to 63.5
  64-65  integrates  63.5  to 65.5
  66-67  integrates  65.5  to 67.5
  68-69  integrates  67.5  to 69.5
  >69    integrates  69.5  to inf

This ensures all probability mass is captured and model_prob sums to 1.0. A sanity check assertion enforces this at runtime.

2026-03-30 — Logger schema expanded to separate event_date from snapshot timing
Original spec used a single 'date' field. This is ambiguous when (a) collector.py takes multiple snapshots per day for the same event, (b) markets for tomorrow's temperature are already open during today's collection window, and (c) different NBM cycles produce different model_prob values for the same event_date. New schema uses event_date (which day's high temperature settles this market), snapshot_ts (when was this data captured), nbm_cycle (which NBM run produced the percentiles), and forecast_lead_hours (how far ahead is the forecast). The morning_model.py 7am run is tagged snapshot_type = 'morning' and is the only data used for v0 evaluation gates.

2026-03-30 — Market structure research integrated into hypothesis framework
Research conducted 2026-03-30 identified: (1) Bürgi et al. (2026) documents favorite-longshot bias across 300k+ Kalshi contracts -- cheap contracts overpriced, expensive contracts yield small positive returns as maker. Bias is present across politics, entertainment, and economic categories. (2) At least five bot types actively trade Kalshi weather markets using real-time NWS data feeds (DSM bot, OMO bot, 6-hour bot, CLI bot, market-maker bot). (3) NWS forecasts carry persistent 1-2F warm bias that market prices anchor to. (4) Weather is approximately 1% of Kalshi volume (~$12-16M/month), meaning thin liquidity and wider spreads. (5) Open-source bots using GFS ensemble data from Open-Meteo with 8% edge threshold have been documented but reportedly stopped working as conditions changed. These findings strengthen the directional prior (FLB) but also raise the bar on information timing -- if bots already correct the bias by 7am, morning edge may not exist.

Does Kalshi API return bracket boundaries as numeric values or only as label strings?
Resolved (morning_model.py, v0). The API returns ticker strings that encode the boundary convention:

- 'T' prefix = tail bracket (strict inequality)
  T62 = strictly less than 62°F (high of exactly 62 does NOT fall here)
  T69 = strictly greater than 69°F (high of exactly 69 does NOT fall here)

- 'B' prefix = between bracket (inclusive of both endpoints)
  B62.5 = between 62 and 63 inclusive (high of exactly 62 DOES fall here)
  B64.5 = between 64 and 65 inclusive

Source: NHIGH contract terms filed with CFTC (kalshi-public-docs.s3.amazonaws.com/contract_terms/NHIGH.pdf). The payout criterion for 'greater than' strictly excludes the threshold value. 'Less than' strictly excludes the threshold value. 'Between' is inclusive.

Since NWS CLI reports integer °F only, the outcome space is discrete integers with no ambiguity. Each integer maps to exactly one bracket.

For the continuous zone interpolation, apply a half-degree continuity correction: bracket '62-63' integrates from 61.5 to 63.5, capturing all probability mass closer to 62 or 63 than to any adjacent integer. This makes boundaries contiguous and probabilities sum to 1.0.

Operational risks
M

Mitigation: Heartbeat file + alert log catches failures within 2 hours. SQLite is append-only so partial data is never corrupted. Missing snapshots leave gaps but do not invalidate the morning model evaluation (which runs separately). For v0, occasional gaps are acceptable -- the intraday data is analysis context, not the primary evaluation instrument. If the VM is unreliable, the morning model still runs from the local machine.
